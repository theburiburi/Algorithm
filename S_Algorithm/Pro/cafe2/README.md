## 1. 데이터 구조의 승리: TreeSet과 ArrayDeque
TreeSet (O(log N)): hurry() API를 위해 매번 정렬하지 않고, 주문이 들어오거나(order), 음료를 받거나(supply), 취소될 때(cancel) 실시간으로 순위를 유지한 것이 시간 초과를 해결한 결정적 요인입니다.

ArrayDeque (O(1)): 각 음료별로 대기열을 관리하여, 특정 음료가 들어왔을 때 전체 주문을 뒤지는 대신 필요한 주문만 즉시 찾아 배정할 수 있었습니다.

## 2. cancel 시 '이미 받은 음료'의 재처리
이 문제의 변별력은 **"이미 배정된 음료를 버리지 않고 재활용하느냐"**였습니다.

단순히 active = false만 하고 끝냈다면 9점에 머물렀을 겁니다.

취소자가 이미 가졌던 음료들을 꺼내서, 대기열의 다음 사람에게 allocate()를 다시 태워 연쇄적으로 주문을 완료시킨 로직이 100점을 만든 핵심입니다.

## 3. 중복 주문 및 Lazy Deletion 처리
한 주문에 같은 음료가 여러 개 있을 때, 대기열에 인덱스를 중복해서 넣음으로써 각 음료의 배정 기회를 보장했습니다.

또한, 대기열(Deque)에서 데이터를 중간에 삭제하는 것은 비용이 큽니다. 이를 직접 지우지 않고 supply 시점에 if(!o.active)로 체크해서 넘기는 Lazy Deletion 기법이 성능을 대폭 끌어올렸습니다.

## 4. 정렬 기준의 무결성 (Tie-breaker)
TreeSet은 compareTo 결과가 0이면 데이터가 덮어씌워지거나 사라집니다.

totalRemaining (남은 개수)

orderTime (주문 순서)

mID (고유 ID)
이 세 가지를 계층적으로 비교하여 모든 주문이 고유한 순위를 갖게 만든 설계가 오답 없는 완벽한 처리를 가능하게 했습니다.
